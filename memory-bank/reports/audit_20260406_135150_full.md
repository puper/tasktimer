## 🔴 确认的漏洞

### CRITICAL-001: Stop/Start 重启时的数据竞争

**位置**: `tasktimer.go:366` (Start 方法), `tasktimer.go:394` (run 方法)  
**严重等级**: CRITICAL  
**CWE**: CWE-362 (Race Condition)  

**问题描述**:

当引擎被停止后重新启动时，存在严重的数据竞争：

```go
// tasktimer.go:366 (Start 方法)
e.stopChan = make(chan struct{})  // 写入 stopChan

// tasktimer.go:394 (run 方法)
case <-e.stopChan:  // 读取 stopChan (无锁保护)
```

**竞态检测器输出**:
```
WARNING: DATA RACE
Write at 0x00c0002a23c8 by goroutine 433:
  tasktimer.(*Engine[...]).Start()
      tasktimer.go:366 +0x80

Previous read at 0x00c0002a23c8 by goroutine 434:
  tasktimer.(*Engine[...]).run()
      tasktimer.go:394 +0xcc
```

**触发条件**:
1. 调用 `engine.Start()` 启动引擎
2. 调用 `engine.Stop()` 停止引擎
3. 立即调用 `engine.Start()` 重新启动引擎
4. 竞态条件：新 goroutine 读取旧的 stopChan，或旧 goroutine 读取新的 stopChan

**影响**:
- 引擎重启不稳定
- 可能导致 "close of closed channel" panic
- 生产环境中可能导致服务崩溃

**失败的测试**:
- `TestComprehensiveAudit/StopRestartSafety` - DATA RACE
- `TestStopRestartPanic` - DATA RACE
- `TestStopRestartWithTasks` - DATA RACE

**修复建议**:

使用 `sync.Once` 确保安全的 channel 关闭和重建：

```go
type Engine[T any] struct {
    // ... 其他字段
    stopChan   chan struct{}
    stopOnce   sync.Once  // 新增：确保只关闭一次
}

func (e *Engine[T]) Stop() {
    e.mu.Lock()
    if !e.running {
        e.mu.Unlock()
        return
    }
    e.running = false
    e.mu.Unlock()

    // 使用 sync.Once 确保只关闭一次
    e.stopOnce.Do(func() {
        close(e.stopChan)
    })
}

func (e *Engine[T]) Start() {
    e.mu.Lock()
    if e.running {
        e.mu.Unlock()
        return
    }
    e.running = true
    e.stopChan = make(chan struct{})
    e.stopOnce = sync.Once{}  // 重置 stopOnce
    e.mu.Unlock()

    go e.run()
}
```

## ❌ 误报分析 (False Positives)

以下发现经过 Code Critic Agent 严格验证，确认为**非 Bug**：

### 1. 并发安全问题 (FALSE POSITIVE)

**原声称**: errorHandler 可能在执行期间被修改

**验证结果**: 
```go
e.mu.RLock()
handler := e.errorHandler  // 读取引用
e.mu.RUnlock()             // 立即释放
handler(task, r)           // 在锁外调用
```

- RLock 保护"读取引用"操作
- Go 函数值不可变，一旦读取就固定
- 这是**正确的并发设计模式**，避免在锁内调用用户代码

**测试验证**: `TestConcurrentSafety` - PASS

---

### 2. 资源泄漏风险 (FALSE POSITIVE)

**原声称**: panic 会导致 workerPool 槽位泄漏

**验证结果**:
```go
e.workerPool <- struct{}{}
defer func() { <-e.workerPool }()  // 第一个 defer

defer func() {  // 第二个 defer
    if r := recover(); r != nil { /* ... */ }
}()
```

- defer 执行顺序是 LIFO，两个 defer 都会执行
- 即使 panic，第一个 defer 保证槽位释放

**测试验证**: `TestWorkerPoolResourceManagement` - PASS

---

### 3. Replace 的原子性 (FALSE POSITIVE)

**原声称**: Replace 先 Delete 再 Add，可能丢失任务

**验证结果**: Replace 的语义是"替换"，如果新任务已过期，立即执行是**预期行为**。

**测试验证**: `TestReplaceSemantics` - PASS

### 4. lastProcessedSlot 并发访问 (已通过)

**原声称**: 并发访问不安全

**验证结果**: 所有访问都在锁保护下，无实际 Bug。
